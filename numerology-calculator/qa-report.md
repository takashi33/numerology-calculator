# QAレポート：数秘電卓（試作版）

## 実施したチェック
- `node --check script.js` による構文エラーチェック → OK
- 計算ロジック（ライフパスナンバー・デスティニーナンバー・チャレンジナンバー・ソウルナンバー・パーソナリティナンバー）を、Node.js上で手計算例と突き合わせて検証
  - 例：1990年7月15日 → ライフパスナンバー 5、チャレンジナンバー [1, 5, 4, 6]（手計算と一致）
  - 例：1978年12月14日 → ライフパスナンバー 33（マスターナンバー。桁をすべて合計する方式に変更後、正しく33になることを確認）
  - 例：1990年7月15日 → ピナクルナンバー [4, 7, 11, 8]（第3ピナクルがマスターナンバー11になることを確認）、2026年のパーソナルイヤーナンバー 5
  - 例：ライフパスナンバー5 → 年齢範囲 [0〜31歳, 32〜40歳, 41〜49歳, 50歳〜]（36−5=31を起点にした標準的な区分と一致することを確認）
  - 不具合修正：ライフパスナンバーがマスターナンバー（例：33）の場合、そのまま「36−33」を計算すると第1期が「0〜3歳」という不自然な範囲になるバグがあった。年齢範囲の計算時だけマスターナンバーも1桁まで還元するよう修正し、LP=33→「0〜30歳」、LP=11→「0〜34歳」と自然な範囲になることを確認
  - 例：名前「Eve」→ 合計14 → カルミックデット・ナンバーとして正しく検出されることを確認
  - 例：名前「Taro Yamada」→ デスティニーナンバー 9、ソウルナンバー 1、パーソナリティナンバー 8（ソウル+パーソナリティの合計値がデスティニー計算時の合計値36と一致することも確認）

## 追記：チャレンジナンバー（基本の3数字）の計算方式を変更
オーナーからのご指摘を受け、「基本の3数字」のチャレンジナンバーを、欧米式（月日をそれぞれ1桁に還元してから差を取る）から、日本の数秘術でよく使われる「未来数」式（月日の桁を全てバラして合計する）に変更した（`app/script.js`・`webapp/index.html`の両方に反映）。
- 例：1978年12月14日 → 月「12」の桁1+2、日「14」の桁1+4 → 合計8 → チャレンジナンバー8（変更前は差の計算で2だった）
- 「チャレンジナンバー 詳細（人生の4つの時期）」の第1〜第4チャレンジは欧米式の差の計算のまま変更していない（オーナーの意向により、基本の3数字のみ変更する範囲とした）。そのため基本の3数字のチャレンジナンバーと詳細セクションの第1チャレンジが異なる値になる場合がある（spec.mdに明記済み）
- Node.js上で計算式を検証：1978-12-14→8、1990-07-15→4、1990-09-29→2（未来数の理論上の最大値20のケース）、1990-09-02→11（マスターナンバーが出るケース）を確認
- CHALLENGE_MEANINGS（一言解説の対応表）に、新たに出現しうる「9」と「11（マスターナンバー）」の解説文を追加

## 追記：新しいタブで「扉を開く」ボタンが反応しない不具合を修正
オーナーから「新しいタブで開くと扉を開くボタンが反応しない」との報告を受け調査。`sessionStorage`への読み込み処理が、ボタンのクリック処理を登録するコードより前に無防備に実行されており、新しいタブ（特にArtifactの共有リンクを直接開いた場合）ではブラウザのプライバシー制限で`sessionStorage`へのアクセス自体が例外を投げることがあり、その場合スクリプト全体が停止してボタンのクリック処理が登録されない状態になっていた。`sessionStorage`の読み込み・書き込みの両方をtry/catchで保護し、アクセスに失敗してもボタンの反応やページの動作自体は続行するよう修正（`webapp/index.html`）。
また、オーナーの要望により画面上の「合言葉」という表記を「コード入力」（または文脈に応じて「コード」）に変更した（`webapp/index.html`・`webapp/invite-card.html`）。

## 追記：招待カードのQRコードを読み取るとClaudeのチャット画面に飛ぶ不具合を修正
オーナーから「QRコードを読み取ったらClaudeのチャット画面に飛んだ」との報告を受け調査。招待カード（`webapp/invite-card.html`）に埋め込まれていたQRコード画像が、意図した数秘電卓のArtifact URL（`https://claude.ai/code/artifact/c44b2f9d-...`）とは別のURL（作成時のチャット画面等）を指すものになっていたと考えられる。
古い画像がどのURLを指していたか手元では確認できなかった（QRデコードツールがローカル環境になかったため）ため、原因の特定よりも修正を優先し、`qrcode`（npmパッケージ、既にローカルにインストール済み）を使って、正しい数秘電卓のArtifact URLを直接エンコードしたQRコード画像を再生成し、招待カードの画像を丸ごと差し替えた。生成後、画像を目視で確認し、3つの位置検出パターンを含む正常な見た目のQRコードであることを確認済み。
